# Mission: Defend PaperLoom In A Java And AI Engineering Interview

## Why
Turn a broadly understood, vibe-coded system into knowledge that can be explained, traced to code, and defended under interview follow-up. The practical goal is to pass Java backend, AI application, and project-design interviews using PaperLoom as credible engineering evidence.

## Success looks like
- Explain PaperLoom clearly in 30 seconds, 2 minutes, and 10 minutes.
- Trace a research request through Vue, Java, Python, storage, retrieval, and citation persistence.
- Answer core Java, JVM, concurrency, Spring, database, cache, messaging, and distributed-system questions using code from this repository.
- Explain Agent, tool calling, RAG, evidence gating, evaluation, and failure handling without hiding behind framework vocabulary.
- Defend important design decisions, tradeoffs, incidents, tests, and possible improvements honestly.
- Diagnose a small project bug or design change during a live coding or system-design follow-up.

## Constraints
- Current understanding is broad rather than implementation-deep.
- Learning must stay tied to the real repository instead of becoming disconnected interview trivia.
- Retrieval practice and mock follow-ups matter more than passive reading.
- Target seniority, interview date, and weekly study time are not yet known.

## Out of scope
- Memorizing every class or dependency.
- Pretending generated code was personally authored line by line.
- Claiming production scale or capabilities that the repository has not demonstrated.
