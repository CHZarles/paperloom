# Consolidate Golden Data And Run Product-Qdrant Evaluations

PaperLoom will use `research/golden-data/` as the only Golden Data management root. Paper Pack
`data_dir` values are relative to the Golden manifest and cannot escape that directory. Stable corpus
assets previously under `data/golden` move to `research/golden-data/corpora/transformer-bert-gpt`.

The Golden `agent-run` command has one corpus path: `JavaCorpusGateway`, the Java Corpus API, Qdrant,
Current Reading Model validation, and MySQL canonical-location hydration. Product failures remain
visible; there is no in-memory backend or compatibility fallback.

Golden paper IDs and installation-specific product paper IDs remain separate. A versioned local map
binds them for product-aligned runs. The selected cases must have complete, one-to-one mappings for
their fixed Paper Pack scope. Java results are translated back to Golden identities before scoring,
and hydrated reads are matched to authored Anchors with the same page and normalized-text rules used
by the offline Harness.

**Considered Options**

- Keep stable corpus files under `data/golden`. Rejected because authoring, scripts, labels, runs, and
  the expanded corpus already live under `research/golden-data`, leaving two ownership roots.
- Keep an in-memory Golden runner beside Qdrant. Rejected because it tests a different retrieval path
  and can hide product integration failures. Historical comparisons remain offline artifacts only.
- Assume Golden and product paper IDs are equal. Rejected because product IDs are generated by each
  installation's upload and persistence path.
- Test Qdrant only with a separate probe script. Rejected as the sole gate because it does not run the
  same Golden cases through the normal Agents SDK runner and scorer.

**Consequences**

- Corpus and authoring paths are easier to inventory, move, and fingerprint.
- Custom manifests cannot load Paper Packs, Cases, or corpus data outside their Golden root.
- Golden runs require a prepared product corpus and a local identity map.
- The runner has no legacy backend selector or in-memory fallback.
- Exact Anchor scoring remains a contract metric rather than a complete answer-accuracy metric.
- Historical dataset fingerprints that included physical paths must be regenerated after the move.
