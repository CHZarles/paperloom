# 论文权限与检索索引方案（通俗版）

日期：2026-07-16

状态：当前开发阶段已实现。

这是一份额外保存的中文通俗版。原技术方案保留不变：
[Paper Access And Retrieval Lifecycle Simplification Proposal](paper-access-and-retrieval-lifecycle-simplification-proposal-2026-07-16.md)。

## 先说结论

认可这个方向。

开发阶段先把论文分成两个空间：

- **个人空间**：普通用户上传的论文，只属于自己。
- **全局论文库**：只有管理员可以上传并发布，所有用户都能看到。

这样可以先去掉组织空间、用户公开上传、多人共同维护论文等复杂情况。

论文权限最终只需要回答一个问题：

```text
这篇论文在我的个人空间里，或者它已经被管理员发布到全局论文库吗？
```

只要答案是“是”，用户就可以访问；否则不能访问。

## 1. 用户实际会看到什么

普通用户看到：

```text
自己的论文 + 管理员发布的全局论文
```

普通用户看不到其他用户的个人论文，也不能把自己的论文改成全局论文。

管理员上传论文时，也先进入管理员自己的空间。论文处理完成后，管理员再点击“发布到全局”。

“上传”和“发布”分成两个动作，规则更清楚，也能避免一篇还没处理好的论文提前出现在全局论文库。

## 2. 同一篇论文会不会重复处理

不会。

系统继续用 PDF 内容生成的 `paper_id` 识别论文。同一份 PDF 即使被多个人上传，底层也只保存和处理一份：

- 一份 PDF；
- 一套解析结果；
- 一套 Reading Model；
- 一套 Qdrant 索引。

不同用户只是各自拥有一条“这篇论文在我的个人空间里”的记录。

因此，“底层数据共用”不等于“所有用户都能看到”。数据可以共用，权限仍然按个人空间和全局发布记录判断。

## 3. 一篇论文什么时候能检索

必须等整个处理流程完成：

```text
PDF 上传完成
-> Parser 解析完成
-> Reading Model 生成完成
-> Qdrant 索引完成并校验通过
-> 可以检索
```

中间任何一步没完成，这篇论文都不能用于检索。

用户仍然可以在个人空间看到它，但页面只能显示“处理中”或“失败”，不能把它加入对话检索范围。

这样做的好处是，检索代码不需要处理“只有一半数据”的情况。

## 4. 普通用户可以做什么

普通用户可以：

- 上传论文到自己的个人空间；
- 查看处理进度；
- 重试首次处理失败的论文；
- 删除自己空间里的论文；
- 检索自己的已完成论文和全局论文。

普通用户不可以：

- 发布全局论文；
- 查看其他用户的个人论文；
- 重新解析一篇已经完成的论文；
- 重建 Qdrant 索引。

已经处理成功的论文，不给普通用户提供“重新解析”按钮。以后出现明确的用户自助修复需求，再单独设计。

## 5. 管理员可以做什么

管理员比普通用户多两类操作：

1. 把自己空间里已经处理完成的论文发布到全局论文库，或者取消发布。
2. 重建单篇论文或全部论文的 Qdrant 索引。

管理员不能直接把某个普通用户的私有论文发布出去。

如果管理员要发布同一份 PDF，应先把它上传到自己的空间。系统会复用已有底层数据，不会重新解析一遍。

## 6. 为什么不再使用组织标签和公开开关

上传时不再让客户端传 `orgTag` 和 `isPublic`。

所有普通上传都按私有论文处理。是否全局可见，只看管理员是否执行了“发布”操作。

这样可以直接避免这些问题：

- 普通用户误把论文公开；
- 每个接口都要判断用户、组织和公开状态；
- 组织成员变化后，历史权限难以解释；
- Qdrant 中还要保存一份容易过期的权限信息。

本轮不做组织论文空间。以后真的需要团队共享时，再增加独立的团队空间，不把它混进当前个人空间规则。

## 7. Qdrant 重建怎么做

开发阶段不考虑单篇论文不停机重建。

重建一篇论文时，直接按下面的流程处理：

```text
暂时禁止检索这篇论文
-> 删除它原来的 Qdrant 数据
-> 根据现有 Reading Model 重新生成索引
-> 校验写入数量
-> 恢复检索
```

重建 Qdrant 不会重新运行 Parser，也不会重新生成 Reading Model。

因此每篇论文只保留一份当前索引，不需要保存新旧两套索引，也不需要做版本切换。

其他没有重建的论文仍然可以正常检索。

## 8. 如何避免重复重建

同一篇论文同一时间只允许一个索引任务。

系统按 `paper_id` 在 MySQL 中抢占任务。已经有任务运行时，第二个请求直接拒绝，不重复消耗 Embedding 和 Qdrant 资源。

全量重建同一时间也只允许一个任务，并且先按顺序一篇一篇处理。

开发阶段先用最容易理解的串行方式。只有实际测量后确认太慢，才考虑有限并发。

## 9. 重建失败怎么办

失败后，这篇论文保持不可检索，并记录错误原因，等待管理员重试。

MySQL 中的 PDF 解析结果和 Reading Model 仍然保留，所以 Qdrant 数据可以重新生成。

不继续使用可能已经过期的旧索引，也不把失败伪装成“搜索不到结果”。

## 10. Qdrant 整体不可用怎么办

直接返回明确错误，不自动切换到 MySQL、BM25、Elasticsearch 或其他检索方式。

原因很简单：不同检索方式会返回不同结果和不同排序。开发阶段保留一条检索路径，更容易发现问题和复现问题。

以后只有在备用检索的输入、输出、排序和错误规则都定义清楚并经过验证后，才考虑回退。

## 11. 删除论文时怎么处理

用户删除论文时，只先删除自己个人空间里的记录。

- 如果其他用户也上传过这篇论文，底层数据继续保留。
- 如果论文已经发布到全局论文库，底层数据继续保留。
- 只有没有任何个人记录，也没有全局发布记录时，才删除 PDF、解析结果、Reading Model 和 Qdrant 数据。

管理员要彻底删除一篇全局论文，必须先取消发布。

## 12. 这个方案省掉了什么

本轮暂时不做：

- 组织或团队论文空间；
- 普通用户公开论文；
- 用户之间共享论文；
- 已完成论文的用户自助重解析；
- 同一篇论文保存多份 Qdrant 索引；
- 新旧索引不停机切换；
- 并行全量重建；
- Qdrant 失败后的自动检索回退。

这不是认为这些需求永远不需要，而是开发阶段先不为没有出现的需求付出复杂度。

## 13. 以后什么时候重新讨论

出现以下情况时，再扩展方案：

- 明确需要团队共享论文：设计独立的团队空间和权限规则。
- 用户经常需要修复已完成论文：设计受控的重新解析流程。
- 全量重建实测太慢：增加有限并发。
- 单篇重建时间影响真实使用：再考虑保留旧索引和不停机切换。
- 业务要求 Qdrant 故障时继续服务：先定义并验证统一的回退规则。

## 最终规则

可以把整套方案记成下面六句话：

1. 普通用户上传的论文只进入自己的个人空间。
2. 只有管理员能把自己上传且处理完成的论文发布到全局。
3. 同一份 PDF 的底层解析结果和 Qdrant 索引只保存一份。
4. Parser、Reading Model 和 Qdrant 全部完成后才能检索。
5. Qdrant 重建时直接暂停该论文，删除旧索引后重新生成。
6. 失败就明确报错并等待重试，不做隐藏回退。
