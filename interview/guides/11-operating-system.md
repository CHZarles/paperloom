## Q1-Q5：操作系统、内核态与用户态

### Q1-Q3 定义

操作系统运行在硬件之上，为应用程序提供进程、内存、文件和 I/O 等抽象，并管理 CPU、内存、磁盘和设备。内核是操作系统中拥有最高权限、负责核心资源管理的部分。面试不要把“操作系统就是内核”说成完整定义：内核是核心，操作系统还包含用户空间工具和服务。

### Q4-Q5 用户态和内核态

用户空间权限较低，不能直接操作硬件；内核空间由内核访问。应用需要文件、Socket、进程或内存服务时，通过系统调用进入内核；硬件中断、异常也会让 CPU 转到内核处理，完成后再返回用户态。

```text
Java/Python 用户线程
        │ read / write / socket / futex
        ▼
系统调用 → 内核处理 → 驱动/缓存/调度
        ▲
        └──────── 返回用户态
```

PaperLoom 的 PDF 读取、HTTP/WebSocket、Kafka/Qdrant/MySQL/Redis 客户端最终都依赖系统调用。性能分析不能只盯 Java 方法耗时，还要看线程阻塞、上下文切换、Page Cache、Socket 和文件描述符。

## Q6-Q15：进程、线程、调度与进程间通信

### Q6 并发和并行

并发是同一时间段内多个任务交替推进；并行是同一时刻由多个 CPU 核真正执行。单核也可以并发，多核才可能把任务并行执行。PaperLoom 中同时存在 Java 服务、Python harness、MinerU、Kafka、Qdrant 等进程；Java 进程内部的线程池并不等于这些任务都在并行，最终还受 CPU、阻塞 IO 和下游容量约束。

### Q7 进程上下文切换

操作系统切换 CPU 所运行的进程时，需要保存当前寄存器、程序计数器和调度状态，再恢复下一个进程的状态。线程过多会增加调度、缓存失效和栈内存开销；阻塞 IO 的线程也会占用线程资源，即使没有持续消耗 CPU。

### Q8-Q10 进程状态、僵尸与孤儿

进程常见状态是新建、就绪、运行、阻塞和退出。阻塞表示等待 IO、锁或其他事件，获得 CPU 也不能继续运行。子进程已退出但父进程尚未 `wait/waitpid` 回收，进程表项仍在，叫僵尸进程；父进程先退出、子进程仍运行，子进程会被 PID 1 的 init/systemd 接管，叫孤儿进程。PaperLoom 的主要部署单元是 Java/Python 服务，不要把 Java 线程状态直接等同 Linux 进程状态。

### Q11 进程调度算法

原书列出 FCFS、短作业优先、优先级、时间片轮转和最短剩余时间优先：FCFS 简单但长任务会阻塞短任务；短作业优先可能饿死长任务；优先级调度需要防止低优先级饥饿；时间片轮转要在响应性与切换开销间平衡；最短剩余时间优先是抢占式版本。面试重点是知道取舍，不要声称自己修改了 Linux 调度器。

### Q12 进程间通信（IPC）

书中方式及边界：

| 方式 | 核心 | 局限/适用 |
| --- | --- | --- |
| 管道 | 内核缓冲的字节流；匿名管道常用于有亲缘关系进程 | 容量有限、语义简单 |
| 信号 | 异步通知进程处理事件 | 携带信息少，不能当业务消息 |
| 消息队列 | 内核中的有界消息链表 | 需要用户态/内核态拷贝 |
| 共享内存 | 多进程映射同一物理内存 | 速度快，但必须另配同步 |
| 信号量 | 计数器，P/V 操作协调资源 | 主要同步，不承载复杂消息 |
| Socket | 本机或跨机器的网络端点 | 有网络、序列化和超时成本 |

PaperLoom 的 Java 与 Python harness 通过 HTTP Socket 通信；Java 与 Kafka、Qdrant、MySQL、Redis 也通过网络协议。Kafka 是独立消息中间件，不等于“Linux 内核消息队列”；项目没有用共享内存把 Java 和 Python 连在一起。

### Q13 进程和线程的联系与区别

进程是资源分配和隔离单位，线程是 CPU 调度单位。同一进程的线程共享代码、堆、打开的文件和 Socket，但各自有栈、寄存器和程序计数器；创建和切换线程通常比进程轻，但线程共享资源也带来竞态。

PaperLoom 的 Java 服务是一个进程内多线程：`ChatSessionRegistry` 的会话结构、HTTP harness 的任务、Trace 写盘线程都会共享堆和文件描述符。跨服务调用则是不同进程之间的网络通信。

### Q14-Q15 线程上下文切换与实现

同一进程内的线程切换可以复用进程地址空间，主要切换线程私有栈、寄存器等；跨进程还要切换地址空间和相关缓存，因此成本更高。原书介绍内核态线程、用户态线程和混合模型；现代 Java 线程通常由 JVM 映射到操作系统线程，阻塞 Socket 读取会占住一个平台线程。不要把 Java `CompletableFuture` 自动说成用户态协程，也不要把每个 Future 说成一个 OS 进程。

## Q16-Q20：同步、死锁、活锁与饥饿

### Q16 线程间如何同步

同步的目标是让共享资源在不同执行交错下仍得到正确结果。临界区是访问共享资源的代码段；锁让同一时刻只有一个线程进入，信号量用计数器表示可用资源。忙等待锁会持续占用 CPU；阻塞锁失败后让出 CPU，由调度器稍后唤醒。

PaperLoom 的代码落点是：`ChatSessionRegistry` 用 `ConcurrentHashMap` 管理会话、`CopyOnWriteArrayList` 保存消息，并在单个 session 上 `synchronized`；`AsyncDiskProductTraceSink` 用有界 `ArrayBlockingQueue` 在请求线程和写盘线程之间传递 Trace。这里没有一个全局锁串行所有会话。

### Q17-Q19 死锁与避免

死锁是多个线程互相持有资源并等待对方释放，任何线程都无法继续。四个必要条件是：互斥、请求并持有、不可剥夺、环路等待。破坏一个条件即可降低风险：一次申请所需资源、申请失败主动释放、统一资源申请顺序，或把临界区缩短。

PaperLoom 的项目回答应具体：会话锁只保护一个 session；数据库任务抢占使用状态条件更新，不长时间持有 Java 锁；涉及多个锁时要固定顺序，避免“先锁 A 再锁 B”和“先锁 B 再锁 A”。不要把数据库锁、Redis 命令和 Java `synchronized` 混成一把锁，也不要声称项目做过死锁压测。

### Q20 活锁与饥饿

饥饿是线程长期拿不到资源，活锁是线程状态不断变化但整体没有进展。重试过于激进、所有线程同步重试或队列长期偏向某类任务，都可能造成类似问题。项目 Kafka 错误处理使用固定 3 秒重试且有最大次数，避免无限重试；当前没有证明实现了公平锁、抖动退避或专门活锁检测，不能夸大。

## Q21-Q29：虚拟内存、分页、置换和链接

### Q21 虚拟内存

虚拟内存把每个进程看到的虚拟地址映射到物理内存，必要时可把不常用页面换到磁盘。它提供隔离和比物理 RAM 更大的地址空间，但会引入页表、TLB、缺页和换入换出成本。

排查 PaperLoom 内存问题时要分层：JVM Heap（对象和 GC）、JVM/Native 内存、进程线程栈、文件映射、OS Page Cache 和 Swap 不是同一块东西。Qdrant 出现 `Too many open files` 首先是 FD/Socket/文件资源上限问题，不应直接归因于 Java Heap 泄漏。

### Q22-Q27 地址转换、分页和交换

- **分段：** 按代码、数据、栈、堆等逻辑段划分，虚拟地址包含段号和段内偏移，段大小不固定。
- **分页：** 把虚拟和物理空间切成固定大小的页，地址由页号和页内偏移组成；页表完成映射，缺页时由内核装入页面。
- **多级页表：** 对页表再次分页，按需创建较低级页表，减少大地址空间下的常驻开销。
- **TLB（快表）：** CPU 中缓存近期页表项，命中可减少地址转换访问；未命中需要查页表。
- **分页和分段区别：** 段是用户可见的逻辑单位、大小不固定；页是系统管理的物理单位、大小固定。
- **交换空间：** 内存紧张时把不常用页面移到磁盘 swap，释放 RAM；Swap 不是“增加了同等速度的内存”，大量换入换出会显著变慢。

### Q28 页面置换算法

| 算法 | 淘汰依据 | 记忆点 |
| --- | --- | --- |
| OPT | 未来最长时间不访问 | 理想算法，无法实际预知未来 |
| FIFO | 最早进入内存 | 简单，但可能淘汰仍常用页 |
| LRU | 最久未被访问 | 利用历史近似未来，维护成本高 |
| Clock | 环形指针和访问位 | LRU 的低成本近似 |
| LFU | 访问次数最少 | 可能保留旧热点，需处理计数衰减 |

这部分是操作系统算法题，项目没有自定义页面置换器。遇到 JVM GC 追问时，不要把 OS 页面置换算法直接当成 G1/ZGC 的垃圾回收算法。

### Q29 硬链接和软链接

硬链接是多个目录项指向同一个 inode，删除一个名字不一定删除数据，不能跨文件系统且通常不能链接目录；软链接是拥有独立 inode 的路径文件，可以跨文件系统，但目标删除后会变成悬空链接。部署脚本、日志目录和挂载路径可能用软链接，但 PaperLoom 业务代码没有实现链接管理，知道语义即可。

## Q30-Q32：零拷贝、阻塞 IO 与多路复用

### Q30 零拷贝

传统文件发送要在磁盘、内核缓冲区、用户缓冲区和 Socket 缓冲区之间多次复制并发生用户态/内核态切换。`mmap + write` 把文件映射到用户空间，`sendfile` 可让内核直接把文件发送到 Socket，目标是减少复制和切换；“零拷贝”不是字面上完全没有任何复制。

Kafka、RocketMQ 等中间件会在 Broker 层利用相关机制，PaperLoom 的 Java 业务代码没有自己实现 `sendfile`、`mmap` 或 Broker 存储。PDF 流、HTTP NDJSON 和 WebSocket 代码不要包装成“零拷贝链路”。

### Q31 阻塞/非阻塞与同步/异步 IO

阻塞/非阻塞描述调用线程是否在数据准备前等待；同步/异步描述数据准备和从内核到用户缓冲区的完成是否需要调用方等待或参与。阻塞、非阻塞轮询、IO 多路复用在经典定义下都可能是同步 IO；真正异步 IO 由内核完成数据搬运后再通知应用。

PaperLoom 的 `PythonResearchHarnessClient` 为每个研究请求安排任务，读取 HTTP response 的 `InputStream`/NDJSON；这是工作线程上的阻塞读取，不是 AIO，也不是一个 Selector 管理所有连接。`CompletableFuture` 只是 Java 的任务编排 API，不会自动把阻塞流变成异步 IO。代价是线程数、超时、取消和文件描述符都要受控。

### Q32 IO 多路复用：select、poll、epoll

IO 多路复用让一个线程管理多个 Socket：

| 机制 | 关注集合 | 就绪处理 | 主要问题 |
| --- | --- | --- | --- |
| select | 固定大小 fd_set | 每次用户/内核来回拷贝并线性扫描 | 常见 FD 上限和 O(n) 扫描 |
| poll | 动态数组 | 仍需线性扫描全部 FD | 无固定 1024 限制，但仍是 O(n) |
| epoll | 内核维护关注对象和就绪队列 | 返回活跃 FD | 适合大量连接、少量活跃事件 |

```text
大量 Socket
    │
    ├─ select/poll：每次把集合交给内核，再扫描全部 FD
    └─ epoll：注册一次，等待就绪事件
```

项目没有自行编写 epoll/Selector 服务端；Spring WebSocket、HTTP 客户端和 Kafka 客户端的底层网络模型由框架负责。面试时把“项目使用 WebSocket”与“项目自己实现 epoll”严格分开。

## 项目版操作系统排障顺序

遇到“服务慢、连接失败、内存高”时，按资源层而不是凭感觉归因：

1. 先区分 CPU 饱和、线程阻塞、网络延迟、磁盘 IO、FD 上限和内存压力。
2. CPU 高看线程数、上下文切换、锁竞争和热点代码；线程越多不一定吞吐越高。
3. 内存高区分 Heap、Native、线程栈、Page Cache、mmap 和 Swap；不要只看 `-Xmx`。
4. 连接异常先查 FD、Socket、连接池和应用层 HTTP 状态；TCP connect 成功不代表 `/healthz` 返回 200。
5. Qdrant 曾出现 `Too many open files`，正确方向是提高 nofile、用 HTTP 200 探活、隔离 Benchmark 实例并持久化部署配置，而不是先调 Java GC。

## 最后背一遍：项目版操作系统自我介绍

> 我会先把操作系统问题分成进程、线程、内存、文件和网络 IO。PaperLoom 是 Java 服务进程，内部有 WebSocket、研究 harness、Kafka Consumer 和 Trace 写盘线程；线程共享堆和文件描述符，所以不能无限扩线程。Python harness 的 NDJSON 是工作线程上的阻塞 InputStream，`CompletableFuture` 不会把它变成 AIO；项目也没有自己实现 epoll 或零拷贝。排障时把 JVM Heap、Native、线程栈、Page Cache、Swap 和 FD 分开看，Qdrant 的 `Too many open files` 先按 FD/Socket 上限和 HTTP 探活处理，不误判成 Java 堆泄漏。死锁方面缩小 Session 锁粒度、避免嵌套锁，数据库任务用条件更新，不长时间持有全局锁。
