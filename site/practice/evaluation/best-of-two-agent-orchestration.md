# 我让 Agent 同时跑两遍，严格评分从 60% 降到了 53.3%

> 当前状态：这个实验没有进入默认运行时。PaperLoom 已恢复为每个研究回合只运行一个 Agent。

<PracticeArticleOverview />

## 背景

前一轮增加提交检查后，Agent 会为缺少引用的论文继续搜索和修改答案。4 个重点问题一共用了约
60.9 万 Token，其中一个问题经历 19 次模型调用、26 次工具调用和 7 次提交拒绝才通过。

长时间在同一段上下文中修补，有两个明显风险：模型会围绕最早找到的位置反复修改，很少重新探索；
每次调用还要重复携带越来越长的历史和工具结果。

实验因此尝试另一种运行方式：同一个问题交给两个彼此隔离的 Agent，各自完成一份答案，再从中选择
一份返回给用户。

文中保留三个实验名称。Sample 表示一次独立运行，Selector 表示最后的选择规则，Oracle pass@2
表示假设每次都能从两份答案中选中通过者时的理论上限。

## 要解决的问题

要验证的假设有三部分：

1. 两次独立运行能否找到不同证据，产生互补答案；
2. 固定选择规则能否挑出较好的一份；
3. 并行执行能否缩短等待时间，同时把总成本控制在可接受范围内。

历史 MiniMax 结果是 `18/30`，但它来自多次运行合并，只能作为参考，不能当作严格的同条件对照。
实验结束后恢复单路径运行，当前代码得到 `17/30`，这个数字更接近可重复的生产基线。

## 怎么处理

### 两次运行完全隔离

两次运行只共享用户问题和只读论文数据。每次运行都单独创建：

- `ResearchRunContext`；
- 论文和位置授权；
- Evidence ID 与已读证据；
- SDK Session 和工具轨迹；
- Model Client；
- 模型调用、工具调用、Token 和提交拒绝计数。

隔离的目的很直接：第二次运行不能看到第一次已经授权的位置、生成的证据编号或会话历史。一次
运行失败也不会自动取消另一次，只有用户取消才结束整组任务。

### 给每次运行固定预算

第一版预算为 5 次模型调用、8 次论文工具调用和 1 次提交修复。4 个重点问题中有 3 个还没形成完整
答案就耗尽预算。模型调用提高到 7 次后，仍有 3 个技术失败。

多论文回答至少要完成以下步骤：

```text
识别论文
-> 分别定位内容
-> 读取多个位置
-> 整理带引用的答案
-> 必要时修正一次提交
```

最终每次运行的上限调整为：

```text
max_model_calls = 10
max_corpus_tool_calls = 12
max_rejected_final_submissions = 2
```

这三个数字分别限制模型调用、论文工具调用和被拒提交次数。

### 用固定规则选择一份答案

两次运行都完成后，固定选择规则按以下顺序比较：

1. 用户明确要求的论文是否都有同论文正文引用；
2. 数值声明能否在引用原文中找到对应数值；
3. 是否讨论了当前问题没有要求的额外论文；
4. 被拒提交、模型调用、工具调用和 Token 是否更少；
5. 完全相同时按固定序号选择。

选择规则不读取测试问题编号、指定证据位置、预期事实、人工标签或自动分数，也不把两份答案拼成
第三份。两次完整运行都会保存，因此离线分析可以同时看到单次结果、理论上限和实际选择结果。

## 遇到的问题

### 多跑一次，没有得到足够多的新答案

30 个固定问题的结果如下：

| 指标 | 结果 |
| --- | ---: |
| 最终选中答案通过严格评分 | `16/30`，`53.3%` |
| 旧 15 个问题 | `13/15`，`86.7%` |
| 新 15 个问题 | `3/15`，`20.0%` |
| 两份答案的理论上限 | `17/30`，`56.7%` |
| 技术失败 | `5/30` |
| 60 次独立运行中预算耗尽 | `16/60` |

两次运行单独得分为：

```text
Sample 0：13/30
Sample 1：14/30
两份答案的理论上限：17/30
Selector 最终结果：16/30
```

第一次运行独有 3 个通过问题，第二次独有 4 个，两者共同通过 10 个。即使每次都能选中通过的那份，
上限也只有 `17/30`，仍低于历史参考 `18/30`。

两次运行使用了相近的检索词，也围绕相近的证据组织答案。运行次数翻倍，探索范围没有同步扩大。

### 选择规则看不到测试指定的证据位置

主要错误集中在证据位置：

| 错误 | 次数 |
| --- | ---: |
| `REQUIRED_ANCHOR_MISSING` | 26 |
| `REQUIRED_ANCHOR_NOT_CITED` | 26 |
| `REQUIRED_PAPER_MISSING` | 8 |
| `OUTCOME_MISMATCH` | 6 |
| `CITATIONS_REQUIRED` | 5 |

模型经常已经引用同一篇论文的相关正文，却没有选中测试数据指定的位置。产品中的选择规则只能检查
论文覆盖、引用内容、数值和成本，无法知道哪段原文正好符合测试要求。

只有 `toolsandbox_constraint_selection_001` 出现明确的选择失误：两份答案中有一份通过，选择规则
选了另一份。修好这一例，结果也只能从 `16/30` 到 `17/30`。主要限制仍然是两次运行没有产生更好
的答案。

### 等待时间缩短，总 Token 明显增加

两次运行并行后，它们的耗时不会直接相加。每个问题的等待时间接近较慢的那次运行，全量
30 个问题用了约 19.1 分钟。

模型账单按两次运行的总用量计算：

| 成本指标 | 全量结果 |
| --- | ---: |
| 模型 HTTP 响应 | `364` |
| 论文工具调用 | `325` |
| Prompt Token | `3,290,152` |
| Completion Token | `147,145` |
| Total Token | `3,437,297` |

研究 Agent 的每一步都要携带问题、会话历史、工具定义和工具结果。并行运行会把大部分上下文成本再
支付一次。本次实验缩短了部分等待时间，没有提高最终结果，也没有降低每个通过答案的成本。

### 全量运行暴露了计费和取消问题

这次实验仍然找出几项单元测试没有覆盖的问题：

- 第一版只统计胜出答案的 Token，漏掉失败或未选中运行的成本；
- 两次运行都失败时，外层技术失败记录丢失了内部用量；
- 用户取消后，另一条运行仍可能继续；
- 胜出运行先结束时，请求完成时间被提前记录；
- 两条进度事件缺少 `sampleIndex`，无法区分来源；
- 历史用户问题中的旧论文会污染 Topic Change；
- `get_research_skill` 被误计入论文工具预算。

原始 `model.response` 事件仍能还原 343.7 万 Token。随后增加了带 diagnostics 的
`HarnessExecutionFailed`，让全部失败时的用量也进入最终记录，并补上取消、时间和事件归属测试。

## 结果

实验没有进入生产。恢复单路径后，同一套当前代码得到：

| 指标 | MiniMax 单路径 |
| --- | ---: |
| Hard Pass | `17/30`，`56.7%` |
| 旧 15 个问题 | `12/15`，`80.0%` |
| 新 15 个问题 | `5/15`，`33.3%` |
| 技术失败 | `1/30` |
| 模型调用 | `212` |
| 工具调用 | `167` |
| Total Token | `2,702,429` |
| 累计问题耗时 | 约 `36.1` 分钟 |

单路径比 Best-of-2 多通过 1 个问题，并少用约 73.5 万 Token。并行运行、选择规则、独立预算和
相关生产代码已经删除。

保留下来的内容有三项：

- 每次运行使用独立状态的实现经验；
- 同时保存所有候选运行，用于区分“没有生成好答案”和“生成后选错”；
- 从原始事件离线计算理论上限、选择损失、成本和错误分布的方法。

`EvalRecorder` 仍只保存：

```text
<run_id>/events.jsonl
<run_id>/result.json
```

这次结果足以停止把 Best-of-2 当作默认提分方案。后续工作回到证据选择、回答类型和人工审核，不再
为同一假设增加并行层数。

随后进行的模型 API 对照见
[`换成 GPT 后，严格评分为什么从 60% 降到 36.7%`](provider-migration-experiment.md)。
